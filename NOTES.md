# 実験ログ / 知見メモ

笛検出の試行錯誤と測定結果を記録する（同じ失敗を繰り返さないため）。

## 正解データ
- 全編リスニングで **笛は計48個**（当初想定「約20」の倍以上）。`labels/ground_truth.tsv`。
- 誤検出のハード負例: 29:55。
- **エネルギーベースでは検出不能な4個**: 1:57, 11:35, 12:39, 23:57（背景に対する突出が小さい／声等に埋もれる）。

## 検出器の性能（`scripts/evaluate.py`, 笛48個, 許容±4s）
アンサンブル(v1∪v2)を合議スコア順にした候補:

| N | precision | recall | F1 |
|---|-----------|--------|-----|
| 15 | 100% | 31% | 48% |
| 30 | 73% | 46% | **56%(最良)** |
| 70 | 44% | 65% | 53% |
| 120 | 36% | 90% | 51% |

- **再現率上限 = 90%**（48個中43個）。v1単独58% / v2単独79% で相補的。
- 上位15件は誤検出ゼロ。
- **F1は56%で頭打ち** = ランキング調整ではこれ以上伸びない（エネルギーの大小では笛と紛らわしい音を分離しきれない）。

## 実験: wav前処理 HPSS（調波・打楽器分離） → ✗ 不採用
時間方向メディアン=調波(笛)、周波数方向メディアン=打楽器(ボール音)を分離し調波を強調した音声を作って評価。

| 設定 | recall上限 | F1最良 |
|------|-----------|--------|
| 元音声(前処理なし) | **90%** | **56%** |
| HPSS KT=7,KF=9 | 69% | 58% |
| HPSS KT=3,KF=13 | 77% | 56% |

**結論: 効果なし（むしろrecall上限が悪化）。**
- 原因1: 音声再構成(ISTFT)が弱い笛を潰し、検出可能だった笛が検出不能になる。
- 原因2: v2が既に「笛帯域の局所背景比」で適応正規化しており、前処理と重複。
- 原因3: 声・チャントも調波構造なのでHPSSでは笛と分離できない（本質問題が残る）。
- 気づき: 調波強調するなら音声に戻さず検出器内部(スペクトログラム上)でやる方が筋が良い。
- スクリプトは `scripts/preprocess.py` に実験用として残置（パイプラインには未組込）。

## 実験: 音色ベース分類器（候補の再ランク） → ◎ 有効
アンサンブル候補(123)に正解で笛/非笛ラベル(正47/負76)を付け、各候補の音色特徴
(onset/倍音比/トーン性/ピッチ安定性/低域比/持続 等11次元)を抽出し、numpyで
ロジスティック回帰を学習。**5分割交差検証(out-of-fold)** で正直に評価。`scripts/classify.py`。

| N | baseline F1 | 分類器 F1 |
|---|------------|-----------|
| 30 | 56% (P73/R46) | **69% (P90/R56)** |
| 40 | 52% (P57/R48) | **70% (P78/R65)** |
| 最良 | **56%** | **72% (N=35)** |

- **F1 56%→72% (+16pt)**。同じrecallをはるかに少ない誤検出で達成。
- 効いた特徴: onset(鋭い立ち上がり)が最強、低域比(群衆の少なさ)、倍音比、持続。
- recall上限は90%のまま(分類器は既存候補の再ランクのみ。検出不能4個は変わらず)。
- ※1試合・正例48での交差検証。他試合での汎化は未検証(要・別試合データ)。

## 教師データ作成ツール (make_review.py / review_server.py)
- 検出器の候補(123)を音声クリップ+スペクトログラム付きHTMLでレビューし、W/N/Uで高速ラベル付け。
- **自動保存**: review_server.py を立てるとラベル変更のたびに /save へPOSTしてファイル保存。
  → エクスポートを忘れても失われない(「export忘れ」問題の対策)。
- 進捗ミニマップ・未/済の色分け・残り件数・Enterで次の未ラベル、で「これから/済」が一目瞭然。

### ラベリングで判明した知見 (正解を 48→47 に修正)
- 候補レビューだけだと笛30/非笛93となり、確定48個と13個食い違った。
- その13個を「正解時刻に中心を合わせた4秒クリップ」で聴き直したら **12個は笛**だった。
  → 原因は **候補の時刻が実際の笛と数秒ずれ**ていて、候補中心のクリップが笛を外していたこと。
  教訓: レビュー用クリップは候補時刻ではなく、近傍のエネルギーピークに中心を合わせるべき。
  → **実装済み**: make_review.py に refine_time(±2.5s内の笛帯域ピークに中心合わせ)を追加。
     近接した別々の笛が統合されないよう衝突回避(1.2s)も実装。クリップも4秒に延長。
- 残り1個 **43:18 は笛でないと確認** → 元の手書き48個の数え過ぎ。正解データから除外。
- **確定: 笛47個 + ハード負例2個(29:55, 43:18)**。

## 汎化検証: 前半で学習→後半でテスト (classify_generalize.py)
後半(2026-l1-final-2nd, 47.5分)を改良レビューツールでラベル付け(候補118個 → 笛41/非笛77)。
前半の笛で学習した音色分類器を、未学習の後半候補に適用:

| 後半での手法 | precision | recall | F1 |
|--------------|-----------|--------|-----|
| baseline(合議スコア順) | 75% | 73% | 74% |
| 分類器(前半で学習→後半に適用) | **92%** | **90%** | **91%** |

- **F1 74%→91%**。音色特徴(onset/倍音比/低域比 等)が試合をまたいで汎化することを実証。
- 注意: 前半/後半は同一試合(同じ主審・笛・録音環境)なので、別主審・別会場への汎化は未検証。
- 注意: 後半の正解は「候補のみ」をラベル付け(全編リスニングなし)なので、後半の上限recallは
  100%(候補=正解の母集合)。絶対recallは未確定。ランキング品質の改善(74→91)は有効な結果。

## 実験: 横リッジ検出（「検出不能4個」の回収可能性） → △ 部分的に有効
発端は 1:57 のスペクトログラムを見て「目には分離可能に見える」という観察。数値で確認:
- 1:57(117.03s, f0=2227Hz)の **2227Hz行**は、同じ行の前後3秒中央値(0.91)に対し中心値6.31で
  **約7倍突出**し、1.5倍閾値超えが**0.79秒(37フレーム)連続** = 細く持続する横線が確かに存在。
- なのにエネルギー検出が外す理由: 検出器は**フレーム単位**で帯域を1スカラーに潰すため、
  117sは帯域ピークで上位3.3%・トーン性で上位25%止まり(群衆が同帯域を広く埋める)。
  **判別情報は瞬間エネルギーでなく「時間方向に連続した細い横線」**にあり、per-frameでは捨てられる。

横リッジ検出器を試作(単発スクリプトで検証): ①各周波数行をその行自身の時間背景(4s)で正規化
→ ②時間0.5s平滑(持続性) → ③フレーム内中央値を引く(周波数の細さ=群衆の広帯域を相殺) → ④帯域最大。

| 検出不能とされた笛 | エネルギー(v1∪v2) | 横リッジ(top-120内の順位) |
|--------------------|-------------------|---------------------------|
| 1:57  | ✗ | **○ rank51** |
| 11:35 | ✗ | **○ rank83** |
| 12:39 | ✗ | ✗ (>200) |
| 23:57 | ✗ | **○ rank121** |

- **「目に分離可能」は正しかった**: 横リッジ追跡で1:57を候補化できる(rank51)。4個中3個を回収。
- 単体ではランクが深く低Nの精度はエネルギー検出に劣る → **置き換えでなく相補**。和集合で
  エネルギーの再現率上限(90%の壁=外す代表がこの4個)を超えられる。
- ※フレーム単位スコア。横リッジを**正式な候補生成器**にするには周波数ビンごとのリッジ追跡＋
  NMSの作り込みが要る(本実験は概念実証)。

## 実験: 横リッジを音色分類器の特徴に追加 → ✗ F1不変(冗長)
`classify.py` の特徴に ridge(候補±0.4sのリッジ最大値)を1本追加。

| | baseline | +ridge特徴 |
|---|---|---|
| F1最良 | 72% (N=45) | **72% (N=45, 全Nで同一)** |

- ridge係数は +0.20(笛らしさ正)で信号自体はあるが、**onset/dur/h_ratioと冗長**で再ランクは不変。
- 教訓: 横リッジの価値は**再ランク(既存候補の並べ替え)でなく候補生成(プール外の4個の回収)**側。
  classify.pyは候補プールを再ランクするだけなので、ここに足しても検出不能4個は救えない。

## 実験: 笛の長さ実測 → 運用条件「長笛優先・短笛は見逃し可」での評価 (classify_long.py)
笛47個の持続長(f0行が局所中央値1.5倍を超える連続長)を実測:
- 分布: min 0.04s / median 0.26s / **max 0.66s**(全部短い。「長笛」でも〜0.5s級)。
- 検出不能4個の内訳: **1:57=0.47s(長め)** / 11:35=0.09s / 12:39=0.06s / 23:57=0.09s。
  → エネルギーが落とす「長い笛」は **1:57 ただ1個**。残り3個は超短(見逃して良い側)。

「長い笛(>=0.4s, 14個)だけを評価対象」に再定義(短笛は recall 分母から除外):

| 手法 | F1最良 | P/R |
|------|--------|-----|
| 全笛が対象(従来) | 72% | — |
| **長笛のみ・既存システムのまま** | **94%** | P96/R93 |
| +持続長を特徴追加 | 86% | ✗悪化(候補の時刻ズレで持続長計測が不安定) |
| +ridge候補120 +持続長 | 91% | ✗プールを薄める |
| +長ridgeのみ30 +持続長 | 93% | △ほぼ同点・超えず |

- **最大の効きは「コード追加」でなく「評価対象の再定義」**: 運用条件に合わせるだけで 72%→94%。
  長い笛は元々クリアに分離でき分類器が得意。新機構は不要だった。
- 長い笛の再現率上限: energy単独 13/14、energy∪ridge **14/14**(1:57回収)。ただし分類器は1:57を
  上位に置けず、precisionを削らないとF1に反映できない。持続長ゲート0.30sなら R100%/P76%(F1 86%)。
  → **「1:57型(マスクされた長笛)を絶対落とせない」運用のときだけ ridge/ゲートを足す**。それ以外は baseline 最良。
- 実装は `scripts/classify_long.py`(実験)。ridge候補生成器 `ridge_candidates`(倍音つき横リッジ+NMS)も同梱。

## 残課題 / 次の一手
- 分類器を本番パイプラインに組込み(モデル保存→新規音声へ適用)。
- 汎化検証のため別試合のデータ＋ラベルが欲しい。
- 検出不能4個(1:57等)は候補に上がらないので、候補生成側(検出器)の感度も別途要改善。
  → **横リッジ検出を候補生成器として実装**(ビンごとリッジ追跡+NMS)し、エネルギー候補と和集合にする
    のが最有力。本実験で1:57含む3個が回収可能と確認済み。
